## Cambios realizados

### 1. `src/infra/agent.rs` — `RUNTIME` hecho `pub(crate)`
Se cambió la visibilidad del runtime global de tokio de `static` a `pub(crate) static` para que otros módulos (`hooks.rs`, `pipeline.rs`) puedan ejecutar operaciones async desde contextos síncronos.

### 2. `src/infra/hooks.rs` — CA4: `run_hook()` migrado a `tokio::process::Command`
La implementación interna de `run_hook()` ahora usa `tokio::process::Command` ejecutado a través del runtime global (`RUNTIME.block_on()`), en lugar de `std::process::Command` directamente. La firma externa se mantiene síncrona para retrocompatibilidad.

### 3. `src/app/pipeline.rs` — CA1, CA2: `process_story()` y `run_real()` migrados a async

#### `process_story()` (CA1)
- Cambiado de `fn` a `async fn`
- Usa `agent::invoke_with_retry(...).await` en lugar de `agent::invoke_with_retry_blocking()`
- Las operaciones git (`snapshot`, `rollback`) se envuelven en `tokio::task::spawn_blocking()` para no bloquear el runtime async

#### `run_real()` (CA2)
- Cambiado de `fn` a `async fn`
- Las llamadas a `process_story(...)` ahora usan `.await` secuencialmente (sin `tokio::spawn`)
- El procesamiento sigue siendo secuencial: una historia después de otra

#### `run()` (punto de entrada)
- Para `dry_run`: sigue llamando a `run_dry()` síncrono
- Para ejecución real: usa `RUNTIME.block_on(run_real(...))` para bloquear hasta que el pipeline async termine

### 4. `src/infra/git.rs` — Sin cambios (CA5)
Las funciones `snapshot()` y `rollback()` mantienen su implementación con `std::process::Command`. Los callers async (`process_story`) las envuelven en `spawn_blocking()`. Los tests de git.rs ya usan `spawn_blocking` donde corresponde.

### Errores de compilación (11 en total):

#### Tipo 1: Tests síncronos que llaman funciones ahora async (8 errores E0599)

**STORY-011 tests en `pipeline.rs`:**
- `story011::process_story_accepts_shared_state` (línea 1887)
- `story011::process_story_blocked_returns_early` (línea 1909)

**Causa:** Estos tests llaman a `process_story(...)` que ahora es `async fn`. Retorna `impl Future<Output = Result<()>>`, no `Result<()>`. Intentan llamar `.is_ok()` sobre el Future en lugar de usar `.await` o `RUNTIME.block_on()`.

**STORY-012 tests en `pipeline.rs`:**

- `story012::run_real_shared_state_reflects_sequential_processing` (línea 2719)

**Causa:** Estos tests llaman a `run_real(...)` que ahora es `async fn`. Retorna `impl Future<Output = Result<RunReport>>`, no `Result<RunReport>`. Intentan llamar `.unwrap()` o `.is_ok()` sobre el Future.

#### Tipo 2: Future no `Send` para `tokio::spawn` (3 errores)

- `story012::process_story_does_not_block_runtime` (línea 2300)

**Causa:** El test usa `tokio::spawn()` sobre una clausura async que llama a `process_story(...).await`. `tokio::spawn` requiere `Future: Send`, pero:
1. `&dyn Workflow` no es `Sync`
2. `Box<dyn AgentProvider>` no es `Send` a través de puntos `.await`
3. `&dyn AgentProvider` no es `Send`

**Posibles soluciones (para el QA):**
- Opción A: Cambiar el test para no usar `tokio::spawn`, usando `tokio::task::spawn_blocking` o simplemente `.await` secuencial
- Opción B: Añadir `Send + Sync` como supertraits de `AgentProvider` y `Workflow`
