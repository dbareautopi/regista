# Decisión: STORY-001 — `from_name()` → `Result` + validate providers

**Fecha**: 2026-05-05
**Actor**: Dev (pi)
**Historia**: STORY-001

---

## Cambios realizados

### 1. `src/infra/providers.rs` — `from_name()` devuelve `Result`

- **Antes**: `fn from_name(name: &str) -> Box<dyn AgentProvider>` con `panic!` en desconocidos.
- **Después**: `fn from_name(name: &str) -> anyhow::Result<Box<dyn AgentProvider>>` con `anyhow::bail!` en desconocidos.

El mensaje de error mantiene el formato descriptivo original: menciona el nombre inválido y los providers válidos.

### 2. Trait `AgentProvider` ahora requiere `Debug`

Se añadió `std::fmt::Debug` como supertrait de `AgentProvider`. Requerido porque `Result::unwrap_err()` necesita `T: Debug` para el mensaje de panic (aunque `anyhow::Error` ya implementa `Debug`, Rust requiere que ambos tipos del `Result` lo implementen).

Se agregó `#[derive(Debug)]` a las 4 structs: `PiProvider`, `ClaudeCodeProvider`, `CodexProvider`, `OpenCodeProvider`.

### 3. `skill_for_role()` usa `.expect()` con mensaje descriptivo

En lugar de propagar el `Result` cambiando la firma (lo que hubiera sido un breaking change mayor), `skill_for_role()` usa `.expect()` con un mensaje claro:
```
"provider inválido en configuración — ejecuta 'regista validate' para diagnosticar"
```
Esto es fail-fast: un provider inválido en config es un error de configuración que `validate` ya detecta.

### 4. Callers actualizados (3 archivos)

| Archivo | Cambio |
|---------|--------|
| `src/app/plan.rs:81` | `from_name(...)` → `from_name(...)?` |
| `src/app/init.rs:251` | `from_name(...)` → `from_name(...)?` |
| `src/app/pipeline.rs:721` | `from_name(...)` → `from_name(...)?` |

Los tres ya retornaban `anyhow::Result`, así que el operador `?` es natural.

### 5. `src/app/validate.rs` — `validate_providers()` implementado

La función (antes un stub) ahora:

1. Recolecta todos los providers únicos (global + 4 roles).
2. Para cada uno, verifica que el binario existe en `PATH` usando búsqueda manual (`std::env::split_paths`).
3. Si el binario no está:
   - **codex** → `Finding::Warning` (CA7: codex puede instalarse con nombres no estándar vía npm).
   - **Otros** → `Finding::Error` (CA6: binario ausente es crítico).
4. Si el nombre del provider no es reconocido por `from_name()` → `Finding::Error`.

Se añadió helper `find_in_path(binary) -> bool` que recorre `PATH` y verifica existencia del archivo (con soporte `.exe` en Windows).

### 6. `validate()` ahora incluye providers (paso 3)

El pipeline de validación ahora es: config → skills → **providers** → stories → dependencies → git.

## Notas para el Reviewer

- `find_in_path()` es intencionalmente manual (sin crate `which`) para evitar añadir una dependencia.
- `skill_for_role()` mantiene firma `String` para minimizar el impacto en callers. El `.expect()` es seguro porque `validate` ya detecta el error antes de que el pipeline llegue a invocar agentes.
- Los tests del QA ya estaban escritos contra la API `Result`; no fue necesario modificar ningún test.
